# Integration Notes: Opus Review Feedback

## Integrating (Critical)

### Issue 1: Feature flag naming mismatch
**Integrating.** The interview explicitly chose "separate flags" (`claude-api`, `claude-cli`). The spec's original `claude` naming is superseded by the interview decision. Updating plan to use consistent naming: `claude-api`, `claude-cli`, `gemini-api`, `gemini-cli`, `openai-api`.

### Issue 2: Retry module contradicts spec
**Integrating — removing retry from orchestrator.** The spec says "No debe hacer retry automatico a nivel de orquestador." The interview added retry, but embedding it in `launch_agents()` violates the spec. Resolution: remove `retry.rs` module and `RetryConfig` from `MagiConfig`. Instead, expose a public `RetryProvider` wrapper struct that users can compose with any `LlmProvider` to add retry logic. This puts retry in the user's control (spec-compliant) while still providing the functionality (interview-compliant). The wrapper is opt-in, not baked into orchestration.

### Issue 8: HashMap non-determinism
**Integrating.** Switch `HashMap<AgentName, _>` to `BTreeMap<AgentName, _>` in `ConsensusResult` (votes, recommendations) and `AgentFactory` (agent_providers). `AgentName` already implements `Ord`, so `BTreeMap` works directly. This ensures deterministic JSON output and reproducible test assertions.

### Issue 13: Missing Eq + Hash derives on AgentName
**Integrating.** Add `Eq` and `Hash` to all enum derives (AgentName, Verdict, Severity, Mode). Required for `BTreeMap`/`HashMap` keys and general correctness.

### Issue 15: LLM response parsing robustness
**Integrating.** Add a `parse_agent_response(raw: &str) -> Result<AgentOutput, MagiError>` method in the orchestrator that handles: (1) code fence stripping (same logic as CLI provider), (2) preamble text before JSON (find first `{` and last `}`), (3) extra fields (serde ignores by default — add `#[serde(deny_unknown_fields)]` is NOT the right choice here, let serde ignore extras). This method is called before `serde_json::from_str`.

### Issue 11: Cancellation safety
**Integrating.** Use `JoinSet` instead of raw `tokio::spawn` in `launch_agents()`. When `JoinSet` is dropped, all spawned tasks are aborted. This provides automatic cleanup if the caller drops the `analyze` future. Document this behavior in the `analyze()` Rustdoc.

## Integrating (Important)

### Issue 3: Logging contradiction
**Integrating with clarification.** `tracing` is not "direct logging" — it requires a subscriber to produce output. Without a subscriber, all tracing calls are no-ops. This aligns with the spec's "usar trait o callback" intent. Adding `tracing` as a non-optional dependency is justified: it's zero-cost when unused, standard in async Rust, and the spec's constraint is about not writing directly to stdout/stderr (which `tracing` does not do). Document this in the plan.

### Issue 6: content.len() is bytes, not characters
**Integrating.** Document explicitly that `max_input_len` is measured in bytes (UTF-8 encoded), which is the natural Rust behavior and appropriate for LLM token estimation (tokens correlate more with bytes than characters).

### Issue 9: Division by zero
**Integrating.** Enforce `min_agents >= 1` in `ConsensusConfig::new()` / `Default`. Return `MagiError::Validation` if `min_agents` is 0.

### Issue 10: build() infallibility
**Integrating.** Change `build(self) -> Magi` to `build(self) -> Result<Magi, MagiError>` for future-proofing. The error case covers `prompts_dir` failures and potential future fallible config validation.

### Issue 12: CLI provider shell safety
**Integrating.** Add explicit documentation noting `tokio::process::Command` bypasses the shell, preventing injection. This is a documentation-only change.

### Issue 17: Display via thiserror
**Integrating.** Clarify that `Display` is generated by `thiserror`'s `#[error("...")]` attributes, not manual impl.

### Issue 19: report.rs vs reporting.rs naming
**Integrating.** Merge `report.rs` (MagiReport struct) into `reporting.rs`. One module for all report-related types and formatting.

### Issue 20: Missing dependency table
**Integrating.** Add a consolidated dependency table with versions, justifications, and feature flags.

## NOT Integrating

### Issue 4: async-trait vs trait_variant
**Not integrating.** `trait_variant` generates a companion trait with Send bounds but still requires the caller to use the generated trait name. `async-trait` is more widely understood, simpler to use, and the heap allocation overhead is negligible for LLM calls that take seconds. The plan's choice is correct.

### Issue 5: Send bound documentation
**Not integrating as plan change.** The `Send + Sync` requirement is already explicit on the trait. The trait's Rustdoc will naturally document this during implementation. No plan-level change needed.

### Issue 7: Unicode normalization for finding dedup
**Not integrating.** The Python original does case-insensitive matching without normalization. Adding `unicode-normalization` introduces a dependency for an edge case (LLMs producing combining characters). Accept as a known limitation, same as the Python original.

### Issue 14: Verdict::effective() return type
**Not integrating.** A separate `EffectiveVerdict` enum adds type complexity for minimal safety gain. The invariant is documented in the spec ("only Approve and Reject are valid outputs") and will be covered by tests.

### Issue 16: ReportConfig::agent_titles redundancy
**Not integrating.** The HashMap in ReportConfig serves as an override mechanism — if the user provides custom agent titles, they take precedence over `AgentName` methods. The `agent_display()` method already falls back to `AgentName` methods. This is intentional two-tier lookup, not redundancy.

### Issue 18: Mode #[non_exhaustive] and prompt handling
**Not integrating.** `Mode` is an internal enum with exactly 3 variants matching the MAGI system design. Making it `#[non_exhaustive]` would force users to add `_ =>` match arms everywhere, which hurts ergonomics. If a new mode is ever added, it's a major feature that warrants a semver bump anyway.

### Issue 21: Verdict serde mapping
**Not integrating as plan change.** Already confirmed correct — lowercase matches Python. Will verify during implementation.

### Issue 22: Error recovery for invalid data
**Not integrating.** Strict rejection is correct for a library. Clamping or "fixing" LLM output would mask problems and violate the spec's validation requirements. Document as explicit design decision.
